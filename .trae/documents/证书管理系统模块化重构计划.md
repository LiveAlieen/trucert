# 证书管理系统模块化重构计划

## 一、系统现状分析

### 1. 现有模块结构
- **核心模块**：`cert_manager.py`, `key_manager.py`, `file_signer.py`, `verifier.py`, `config.py`
- **GUI模块**：`main_window.py`, `key_tab.py`, `cert_tab.py`, `file_tab.py`, `verify_tab.py`
- **配置和存储**：`configs/`, `configs/key/`, `configs/trust/`, `root_key/`

### 2. 重构后的模块结构
- **核心业务逻辑**：`core/business/`
  - `cert_manager.py`：证书管理
  - `key_manager.py`：密钥管理
  - `file_signer.py`：文件签名
  - `verifier.py`：验证功能
  - `config.py`：配置管理
  
- **服务层**：`core/services/`
  - `cert_service.py`：证书服务
  - `key_service.py`：密钥服务
  - `file_signer_service.py`：文件签名服务
  - `verifier_service.py`：验证服务
  - `config_service.py`：配置服务
  
- **存储层**：`core/storage/`
  - `key_storage.py`：密钥存储
  - `cert_storage.py`：证书存储
  - `config_storage.py`：配置存储
  - `storage_manager.py`：存储管理器
  
- **工具类**：`core/utils/`
  - `crypto_utils.py`：加密工具
  - `file_utils.py`：文件工具
  - `hash_utils.py`：哈希工具
  - `log_utils.py`：日志工具
  - `verify_utils.py`：验证工具
  
- **GUI模块**：`gui/`
  - `main_window.py`：主窗口
  - `key_tab.py`：密钥管理标签页
  - `cert_tab.py`：证书管理标签页
  - `file_tab.py`：文件签名标签页
  - `verify_tab.py`：验证标签页
  
- **配置和存储**：`configs/`, `configs/key/`, `configs/trust/`, `root_key/`

### 3. 重构后的依赖关系分析
- **GUI模块** → **服务层**：GUI通过服务层调用业务逻辑
- **服务层** → **核心业务逻辑**：服务层封装业务逻辑，提供统一接口
- **核心业务逻辑** → **存储层**：业务逻辑通过存储层访问数据
- **核心业务逻辑** → **工具类**：业务逻辑使用工具类提供的功能
- **服务层** → **工具类**：服务层也可以直接使用工具类
- **存储层** → **工具类**：存储层使用工具类处理数据

### 4. 重构前的依赖关系分析
- **GUI模块** 直接依赖 **核心模块**
- **核心模块** 内部存在交叉依赖（如 `verifier.py` 导入 `file_signer.py`）
- **配置管理** 分散在各个模块中
- **存储逻辑** 与业务逻辑耦合

## 二、重构目标

1. **模块化**：明确模块边界，实现高内聚低耦合
2. **分层架构**：业务逻辑、数据访问、界面展示分离
3. **代码优化**：移除重复代码，统一命名规范
4. **可维护性**：完善文档，提升代码可读性
5. **可扩展性**：建立清晰的接口，支持功能扩展

## 三、重构计划

### 1. 模块边界和接口定义

#### 核心模块重构
- **`cert_manager/`**：证书管理模块
  - 职责：证书生成、验证、存储、加载
  - 接口：`generate_cert()`, `verify_cert()`, `save_cert()`, `load_cert()`
  
- **`key_manager/`**：密钥管理模块
  - 职责：密钥生成、存储、加载
  - 接口：`generate_key()`, `save_key()`, `load_key()`
  
- **`file_signer/`**：文件签名模块
  - 职责：文件签名、验证
  - 接口：`sign_file()`, `verify_file()`, `batch_sign()`
  
- **`verifier/`**：验证模块
  - 职责：证书和文件签名验证
  - 接口：`verify_cert()`, `verify_file()`
  
- **`config/`**：配置模块
  - 职责：配置管理
  - 接口：`get_config()`, `set_config()`, `reload_config()`
  
- **`storage/`**：存储模块（新增）
  - 职责：统一数据存储和加载
  - 接口：`save()`, `load()`, `delete()`, `list()`
  
- **`utils/`**：工具模块
  - 职责：通用工具函数
  - 接口：`hash_utils`, `crypto_utils`, `file_utils`

#### GUI模块重构
- **`gui/`**：界面模块
  - 职责：用户交互
  - 接口：通过事件与核心模块通信

### 2. 分层架构设计

#### 三层架构
1. **表示层（Presentation Layer）**
   - GUI模块：`gui/`
   - 负责用户交互和界面展示

2. **业务逻辑层（Business Logic Layer）**
   - 核心功能模块：`cert_manager/`, `key_manager/`, `file_signer/`, `verifier/`
   - 负责业务逻辑实现

3. **数据访问层（Data Access Layer）**
   - 存储模块：`storage/`
   - 负责数据的存储和加载

#### 依赖方向
- 表示层 → 业务逻辑层 → 数据访问层
- 禁止反向依赖

### 3. 模块间通信机制

#### 事件驱动架构
- 使用事件机制实现模块间通信
- 核心模块通过事件通知GUI模块状态变化
- GUI模块通过事件触发核心模块操作

#### 接口调用
- 业务逻辑层通过定义的接口调用数据访问层
- 表示层通过定义的接口调用业务逻辑层

### 4. 代码优化和规范

#### 代码结构优化
- **移除重复代码**：提取通用功能到工具模块
- **统一命名规范**：采用一致的命名风格
- **优化文件结构**：按功能组织文件和目录

#### 命名规范
- **模块名**：小写蛇形命名法（如 `key_manager`）
- **类名**：驼峰命名法（如 `KeyManager`）
- **函数名**：小写蛇形命名法（如 `generate_key`）
- **变量名**：小写蛇形命名法（如 `private_key`）

### 5. 文档和测试

#### 文档编写
- **模块文档**：每个模块编写详细的文档，说明功能和使用方法
- **接口说明**：每个接口编写详细的说明，包括参数、返回值和使用示例
- **架构文档**：编写系统架构文档，说明模块间的关系和依赖

#### 测试计划
- **单元测试**：测试每个模块的核心功能
- **集成测试**：测试模块间的交互
- **系统测试**：测试整个系统的功能和性能
