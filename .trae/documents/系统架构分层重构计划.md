# 系统架构分层重构计划

## 一、重构目标

1. **实现服务层与工具层的明确分离与解耦**：
   - 服务层专注于对外提供标准化接口调用，负责从业务层调用和封装功能
   - 工具层停止直接对接服务层，改为通过业务层进行间接调用

2. **统一接口规范**：
   - 统一GUI和CLI的接口规范
   - 预留Web API接口和未来外部程序调用的扩展接口
   - 所有接口遵循统一的设计标准，严格控制接口数量

3. **确保系统各组件间交互的一致性和可维护性**：
   - 修改现有GUI与CLI程序，使其符合统一后的接口规范

## 二、重构架构设计

### 1. 分层架构

```
+-------------------------+
|        界面层 (GUI)      |
|-------------------------|
|         命令行接口 (CLI)  |
|-------------------------|
|     Web API 接口层       |
|-------------------------|
|       服务层 (Services)  |  # 专注于接口调用和业务逻辑封装
|-------------------------|
|     业务逻辑层 (Business) |  # 核心业务逻辑，调用工具层
|-------------------------|
|      工具层 (Utils)      |  # 提供通用工具功能，不直接对接服务层
|-------------------------|
|      存储层 (Storage)    |  # 数据持久化
+-------------------------+
```

### 2. 接口规范设计

#### 2.1 服务层接口规范

| 服务模块 | 核心接口 | 功能描述 |
|---------|---------|----------|
| CertService | generate_self_signed_cert() | 生成自签名证书 |
|            | generate_secondary_cert() | 生成二级证书 |
|            | save_cert() | 保存证书 |
|            | load_cert() | 加载证书 |
|            | list_certs() | 列出所有证书 |
|            | delete_cert() | 删除证书 |
|            | import_cert() | 导入证书 |
| KeyService | generate_rsa_key() | 生成RSA密钥对 |
|            | generate_ecc_key() | 生成ECC密钥对 |
|            | list_keys() | 列出所有密钥 |
|            | load_key_pair() | 加载密钥对 |
|            | delete_key() | 删除密钥 |
|            | save_private_key() | 保存私钥 |
|            | save_public_key() | 保存公钥 |
|            | load_private_key() | 加载私钥 |
|            | load_public_key() | 加载公钥 |
| FileSignerService | sign_file() | 签名文件 |
|                  | batch_sign() | 批量签名文件 |
|                  | verify_file_signature() | 验证文件签名 |
| VerifierService | verify_cert_signature() | 验证证书签名 |
|                | verify_file_signature() | 验证文件签名 |
|                | verify_cert_chain() | 验证证书链 |
| ConfigService | get_config() | 获取配置 |
|              | set_config() | 设置配置 |
|              | load_config() | 加载配置 |
|              | save_config() | 保存配置 |

#### 2.2 Web API接口规范

- 采用RESTful API设计风格
- 支持JSON格式的请求和响应
- 预留标准的API路径，如 `/api/v1/certs`, `/api/v1/keys`, `/api/v1/signatures` 等

### 3. 代码重构计划

#### 3.1 重构服务层

1. **修改服务层模块**：
   - 移除服务层中直接调用工具层的代码
   - 确保服务层只调用业务逻辑层的接口
   - 统一接口参数和返回值格式

2. **添加Web API接口支持**：
   - 创建Web API模块，提供RESTful接口
   - 确保Web API接口与服务层接口一致

#### 3.2 重构业务逻辑层

1. **修改业务逻辑层模块**：
   - 确保业务逻辑层调用工具层的功能
   - 提供统一的接口给服务层调用

2. **优化工具层调用**：
   - 确保工具层只被业务逻辑层调用，不直接被服务层调用

#### 3.3 修改GUI和CLI

1. **修改GUI模块**：
   - 更新GUI标签页，使其使用统一后的服务层接口
   - 确保GUI操作与CLI操作一致

2. **修改CLI模块**：
   - 更新CLI命令，使其使用统一后的服务层接口
   - 确保CLI操作与GUI操作一致

#### 3.4 测试和验证

1. **单元测试**：
   - 测试服务层接口的正确性
   - 测试业务逻辑层的功能
   - 测试GUI和CLI的操作一致性

2. **集成测试**：
   - 测试整个系统的集成功能
   - 测试Web API接口的可用性

## 四、预期成果

1. **清晰的分层架构**：
   - 服务层与工具层明确分离
   - 依赖关系清晰，易于维护

2. **统一的接口规范**：
   - GUI和CLI使用统一的接口
   - 预留Web API接口和扩展接口

3. **可维护性和可扩展性**：
   - 系统各组件间交互一致

4. **良好的性能**：

## 五、风险评估

1. **兼容性风险**：
   - 重构可能影响现有功能的兼容性
   - 应对措施：详细测试，确保所有现有功能正常工作

2. **性能风险**：
   - 重构可能影响系统性能
   - 应对措施：性能测试，确保系统性能不低于重构前

3. **时间风险**：
   - 重构可能需要较长时间
   - 应对措施：合理规划时间，分阶段实施重构

4. **技术风险**：
   - 新的接口设计可能存在问题
   - 应对措施：充分设计和测试，确保接口设计合理

## 六、结论

通过本次重构，系统将实现服务层与工具层的明确分离与解耦，统一GUI和CLI的接口规范，预留Web API接口和扩展接口，提高系统的可维护性和可扩展性。重构后的系统将更加清晰、灵活，能够更好地满足未来的需求。
